# 05 Message Protocol

## 一、目的

本文档定义：

天长麻将服务器与客户端之间的全部消息协议。

包括：

- 请求（Request）
- 响应（Response）
- 广播（Broadcast）
- 状态同步（Synchronization）

所有客户端必须严格按照本协议实现。

所有服务器不得发送协议之外的消息。

---

## 二、设计原则

### 2.1 权威服务器（Authoritative Server）

服务器拥有唯一游戏状态。

客户端不得自行计算：

- 是否可碰
- 是否可杠
- 是否可胡
- 花数
- 剩余牌数

客户端仅负责：

- 用户输入
- 动画
- UI 显示

所有业务逻辑均由服务器计算。

---

### 2.2 消息驱动（Message Driven）

客户端所有操作：

必须发送消息。

服务器处理后：

返回：

- Response
- Broadcast

客户端不得修改游戏状态。

---

### 2.3 单向状态同步

任何时刻：

服务器状态为唯一真实状态。

客户端发生异常：

例如：

- 网络延迟
- 重连
- 卡顿

服务器同步状态覆盖客户端状态。

---

### 2.4 幂等性（Idempotency）

同一消息：

重复发送。

服务器不得重复执行。

例如：

连续点击：

碰。

服务器只能碰一次。

---

### 2.5 顺序保证（Order）

服务器保证：

所有广播：

严格按照游戏顺序发送。

例如：

摸牌

↓

补花

↓

再次摸牌

↓

可胡

↓

出牌

不得改变顺序。

---

## 三、消息命名规范

统一采用：

MSG-XXX

例如：

MSG-001

MSG-002

MSG-003

……

所有事件名称：

统一使用：

全大写。

例如：

DRAW

DISCARD

PENG

AN_GANG

BU_GANG

TING

WIN

SETTLEMENT

# 四、统一消息字段规范

所有消息必须遵守统一字段规范。

## 4.1 客户端请求基础字段

所有客户端请求必须包含以下字段：

```json
{
  "messageId": "uuid",
  "type": "MESSAGE_TYPE",
  "roomId": "123456",
  "gameId": "game_000001"
}
```

字段说明：

```text
messageId：

消息唯一 ID，用于防止重复请求。

type：

消息类型。

roomId：

房间 ID。

gameId：

当前对局 ID。
```

---

## 4.2 服务器广播基础字段

所有服务器广播必须包含以下字段：

```json
{
  "type": "MESSAGE_TYPE",
  "roomId": "123456",
  "gameId": "game_000001",
  "timestamp": 1782893021
}
```

字段说明：

```text
timestamp：

服务器发送时间。

用于日志、回放、问题排查。
```

---

## 4.3 幂等处理

服务器必须根据：

```text
messageId
```

判断重复消息。

同一个 messageId 不得重复执行。

例如：

玩家连续点击碰牌。

服务器只允许成功处理一次。

---

## 4.4 房间与对局区分

roomId 表示房间。

gameId 表示当前这一局。

同一个 roomId 下可以存在多局 gameId。

例如：

```text
roomId = 123456

gameId = game_000001
gameId = game_000002
gameId = game_000003
```

以后战绩、回放、结算全部以 gameId 为准。

# 五、房间生命周期

## MSG-001 ROOM_CREATE（创建房间）

消息方向：

客户端 → 服务器

目的：

创建一个新的麻将房间。

触发条件：

玩家点击：

创建房间。

请求：

```json
{
  "type": "ROOM_CREATE"
}
```

服务器响应：

```json
{
  "type": "ROOM_CREATE_SUCCESS",
  "roomId": "123456"
}
```

广播：

无。

下一状态：

WAIT_JOIN

说明：

创建成功后。

房主自动进入房间。

服务器成为该房间唯一管理者。

---

## MSG-002 ROOM_JOIN（加入房间）

消息方向：

客户端 → 服务器

目的：

加入指定房间。

触发条件：

玩家输入房间号。

请求：

```json
{
  "type": "ROOM_JOIN",
  "roomId": "123456"
}
```

服务器响应：

```json
{
  "type": "ROOM_JOIN_SUCCESS",
  "seat": 2
}
```

广播：

```json
{
  "type": "PLAYER_JOIN",
  "seat": 2
}
```

下一状态：

WAIT_READY

说明：

服务器分配座位。

所有玩家同步最新房间信息。

---

## MSG-003 PLAYER_READY（玩家准备）

消息方向：

客户端 → 服务器

目的：

玩家确认准备开始游戏。

触发条件：

玩家点击：

准备。

请求：

```json
{
  "type": "PLAYER_READY"
}
```

服务器响应：

```json
{
  "type": "PLAYER_READY_SUCCESS"
}
```

广播：

```json
{
  "type": "PLAYER_READY",
  "seat": 2
}
```

下一状态：

WAIT_GAME_START

说明：

当四位玩家全部准备后。

服务器自动进入：

GAME_START。

---

## MSG-004 GAME_START（开始游戏）

消息方向：

服务器 → 全体客户端

目的：

通知游戏正式开始。

触发条件：

四位玩家全部准备完成。

广播：

```json
{
  "type": "GAME_START",
  "dealer": 0,
  "round": 1
}
```

下一状态：

DEAL

说明：

服务器确定：

- 庄家
- 骰子结果（如需要）
- 牌墙初始化
- 花牌初始化

随后立即进入：

发牌流程。

---

# 六、游戏生命周期

## MSG-005 DEAL（发牌）

消息方向：

服务器 → 全体客户端

目的：

通知玩家发牌完成。

触发条件：

游戏开始。

服务器完成洗牌、发牌。

广播：

```json
{
  "type": "DEAL",
  "dealer": 0,
  "handTileCount": 13
}
```

说明：

服务器分别向四位玩家发送各自手牌。

其他玩家不得收到任何手牌信息。

随后进入：

补花流程。

下一状态：

BU_HUA

---

## MSG-006 BU_HUA（补花）

消息方向：

服务器 → 指定客户端

目的：

通知玩家补花。

触发条件：

玩家手牌存在花牌。

Payload：

```json
{
  "type":"BU_HUA",
  "flowerCount":2
}
```

服务器响应：

服务器完成：

- 花牌移出手牌
- 从牌尾补牌

广播：

```json
{
  "type":"PLAYER_BU_HUA",
  "seat":1,
  "flowerCount":2
}
```

说明：

其他玩家只能知道：

玩家补了几张花。

不得知道：

补到了什么牌。

若继续摸到花牌。

服务器继续执行补花。

直到没有花牌。

下一状态：

DRAW

---

## MSG-007 DRAW（摸牌）

消息方向：

服务器 → 指定客户端

目的：

通知玩家摸牌。

触发条件：

轮到当前玩家摸牌。

Payload：

```json
{
  "type":"DRAW",
  "tile":"5条",
  "remainTiles":72
}
```

广播：

无。

说明：

摸到的牌。

只有当前玩家可见。

服务器随后立即计算：

- 是否可胡
- 是否可暗杠
- 是否可补杠
- 是否可报听

下一状态：

WAIT_ACTION

---

## MSG-008 WAIT_ACTION（等待玩家操作）

消息方向：

服务器 → 指定客户端

目的：

通知客户端当前允许执行的操作。

触发条件：

摸牌结束。

Payload：

```json
{
  "type":"WAIT_ACTION",
  "actions":[
      "WIN",
      "AN_GANG",
      "BU_GANG",
      "TING",
      "DISCARD"
  ]
}
```

说明：

服务器只发送：

当前真正允许执行的操作。

客户端不得自行计算。

下一状态：

等待玩家操作。

---

## MSG-009 DISCARD（出牌）

消息方向：

客户端 → 服务器

目的：

玩家打出一张手牌。

触发条件：

服务器当前状态为：

WAIT_DISCARD。

请求：

```json
{
  "type": "DISCARD",
  "tile": "5条"
}
```

服务器响应：

```json
{
  "type": "DISCARD_SUCCESS"
}
```

广播：

```json
{
  "type": "PLAYER_DISCARD",
  "seat": 1,
  "tile": "5条"
}
```

下一状态：

WAIT_RESPONSE

说明：

服务器必须验证：

- 当前是否轮到该玩家出牌。
- 玩家是否拥有该牌。
- 报听后是否允许打出该牌。

服务器完成出牌后：

立即检查其他三家是否具有响应权。

响应优先级由服务器统一判定。

客户端不得自行计算。

---

## MSG-010 PENG（碰牌）

消息方向：

客户端 → 服务器

目的：

玩家请求碰牌。

触发条件：

上一位玩家打出一张牌。

服务器判定当前玩家允许碰牌。

请求：

```json
{
  "type": "PENG"
}
```

服务器响应：

```json
{
  "type": "PENG_SUCCESS"
}
```

广播：

```json
{
  "type": "PLAYER_PENG",
  "seat": 2,
  "tile": "5条"
}
```

下一状态：

WAIT_DISCARD

说明：

服务器必须验证：

- 玩家拥有两张相同牌。
- 当前牌允许碰牌。
- 玩家当前未处于禁止碰牌状态。

碰牌成功后：

- 三张牌加入公开碰牌组。
- 当前玩家立即获得出牌权。
- 必须继续打一张牌。

客户端仅根据服务器广播刷新牌桌。

不得自行修改牌局状态。

---

## MSG-011 MING_GANG（明杠）

消息方向：

客户端 → 服务器

目的：

玩家请求明杠。

触发条件：

其他玩家刚打出一张牌。

服务器判定当前玩家允许明杠。

请求：

```json
{
  "type": "MING_GANG"
}
```

广播：

```json
{
  "type": "PLAYER_MING_GANG",
  "seat": 2,
  "tile": "5条"
}
```

下一状态：

GANG_DRAW

说明：

服务器必须验证：

- 玩家手中拥有三张相同牌。
- 当前打出的牌允许明杠。
- 玩家当前允许执行明杠。

明杠成功后：

- 四张牌组成公开杠牌组。
- 更新杠牌信息。
- 进入杠尾摸牌流程。

若玩家已经报听：

服务器必须验证杠后仍保持听牌。

否则不得允许明杠。

客户端仅根据广播更新界面。

不得自行计算是否允许明杠。

---

## MSG-012 AN_GANG（暗杠）

消息方向：

客户端 → 服务器

目的：

玩家请求暗杠。

触发条件：

服务器判定玩家拥有四张相同牌。

请求：

```json
{
  "type": "AN_GANG",
  "tile": "5条"
}
```

广播：

```json
{
  "type": "PLAYER_AN_GANG",
  "seat": 1
}
```

下一状态：

GANG_DRAW

说明：

服务器必须验证：

- 玩家拥有四张相同牌。
- 当前允许暗杠。
- 报听后暗杠不得导致失听。

广播时：

不得向其他玩家发送具体牌值。

其他玩家只能知道：

某位玩家完成了一次暗杠。

暗杠成功后：

服务器进入杠尾摸牌流程。

客户端仅根据广播刷新公开信息。

不得推断暗杠牌面。

---

## MSG-013 BU_GANG（补杠）

消息方向：

客户端 → 服务器

目的：

玩家请求补杠。

触发条件：

玩家已经完成碰牌。

随后摸到第四张相同牌。

服务器判定允许补杠。

请求：

```json
{
  "type": "BU_GANG",
  "tile": "5条"
}
```

广播：

```json
{
  "type": "PLAYER_BU_GANG",
  "seat": 1,
  "tile": "5条"
}
```

下一状态：

GANG_DRAW

说明：

服务器必须验证：

- 玩家已经存在对应碰牌组。
- 当前摸到第四张相同牌。
- 玩家允许执行补杠。
- 若玩家已报听，补杠后必须仍保持听牌。

补杠成功后：

- 原碰牌组升级为杠牌组。
- 更新杠牌信息。
- 从杠尾摸牌。

本游戏不支持抢杠胡。

客户端不得自行升级碰牌为杠牌。

必须等待服务器广播。

---

## MSG-014 TING（报听）

消息方向：

客户端 → 服务器

目的：

玩家请求报听。

触发条件：

服务器判定当前玩家允许报听。

请求：

```json
{
  "type": "TING"
}
```

广播：

```json
{
  "type": "PLAYER_TING",
  "seat": 1
}
```

下一状态：

WAIT_DISCARD

说明：

服务器必须验证：

- 当前允许报听。
- 玩家报听后仍保持合法听牌状态。

报听成功后：

服务器更新玩家状态：

```text
isTing = true
```

广播时：

不得向其他玩家发送：

- 听哪些牌。
- 听几张牌。
- 听牌类型。

其他玩家只能知道：

该玩家已经报听。

后续所有操作：

服务器必须按照报听规则进行限制。

客户端不得自行判断是否允许报听。

不得自行计算听牌结果。

所有听牌判断均由服务器完成。

---

## MSG-015 WIN（胡牌）

消息方向：

客户端 → 服务器

目的：

玩家请求胡牌。

触发条件：

服务器已判定当前玩家允许胡牌。

请求：

```json
{
  "type": "WIN"
}
```

广播：

```json
{
  "type": "PLAYER_WIN",
  "seat": 1
}
```

下一状态：

SETTLEMENT

说明：

服务器必须验证：

- 当前玩家是否允许胡牌。
- 当前胡牌是否为自摸。
- 当前手牌是否满足合法胡牌条件。
- 玩家是否处于允许胡牌状态。

胡牌成功后：

- 本局立即结束。
- 服务器进入结算流程。
- 所有玩家停止继续操作。
- 客户端播放胡牌动画。

WIN 消息只负责通知：

谁胡牌。

不得包含：

- 花数。
- 牌型。
- 结算金额。
- 详细计分。

这些内容必须由：

SETTLEMENT

消息单独发送。

---

## MSG-016 SETTLEMENT（结算）

消息方向：

服务器 → 全体客户端

目的：

发送本局完整结算结果。

触发条件：

玩家胡牌成功。

服务器完成花数计算。

广播：

```json
{
  "type": "SETTLEMENT",
  "winnerSeat": 1,
  "baseType": "单吊",
  "patterns": [
    "豪华七对"
  ],
  "modifiers": [
    "清一色"
  ],
  "events": [
    "海底捞月"
  ],
  "bonus": [
    "暗绝"
  ],
  "flower": 107,
  "unitPrice": 5,
  "payPerPlayer": 535,
  "totalWin": 1605
}
```

下一状态：

ROUND_END

说明：

SETTLEMENT 必须包含完整结算信息。

字段说明：

```text
winnerSeat：

胡牌玩家座位。

baseType：

基础胡型。

patterns：

特殊牌型数组。

modifiers：

加成项数组。

events：

胡牌事件数组。

bonus：

奖励项数组。

flower：

最终花数。

unitPrice：

每张花金额。

payPerPlayer：

每名输家应支付金额。

totalWin：

胡牌玩家本局总收入。
```

结算规则：

```text
payPerPlayer = flower × unitPrice

totalWin = payPerPlayer × 3
```

客户端收到 SETTLEMENT 后：

- 展示胡牌玩家。
- 展示牌型。
- 展示事件。
- 展示奖励。
- 展示花数。
- 展示每家支付金额。
- 展示总收入。
- 允许查看计分明细。
- 允许查看上一局牌谱。

SETTLEMENT 是客户端展示结算页的唯一数据来源。

客户端不得自行计算花数或金额。

# 七、系统消息

## MSG-017 TRUSTEE（托管）

消息方向：

客户端 → 服务器

目的：

玩家开启或关闭托管。

触发条件：

玩家主动点击：

托管。

或：

客户端断线超时。

服务器自动进入托管。

请求：

```json
{
  "type": "TRUSTEE",
  "enable": true
}
```

广播：

```json
{
  "type": "PLAYER_TRUSTEE",
  "seat": 1,
  "enable": true
}
```

下一状态：

保持当前游戏状态。

说明：

服务器更新：

```text
isTrustee = true
```

托管期间：

服务器自动执行：

- 摸牌
- 出牌
- 可胡自动胡
- 可补花自动补花

服务器不得主动：

- 报听
- 明杠
- 暗杠
- 补杠

（除非以后规则明确允许。）

玩家重新连接成功后：

服务器立即取消托管。

广播：

```json
{
  "type": "PLAYER_TRUSTEE",
  "seat": 1,
  "enable": false
}
```

## MSG-018 RECONNECT（断线重连）

消息方向：

客户端 → 服务器

目的：

恢复当前牌局。

触发条件：

玩家重新进入房间。

请求：

```json
{
  "type": "RECONNECT"
}
```

服务器响应：

```json
{
  "type": "RECONNECT_STATE",
  "roomId": "123456",
  "gameState": "WAIT_DISCARD",
  "dealer": 0,
  "currentSeat": 2,
  "remainTiles": 56,
  "handTiles": [],
  "flowers": [],
  "melds": [],
  "discards": [],
  "isTing": true,
  "isTrustee": false
}
```

下一状态：

恢复服务器当前状态。

说明：

服务器必须重新同步：

- 当前手牌
- 花牌
- 碰牌
- 杠牌
- 弃牌
- 当前操作玩家
- 剩余牌数
- 当前允许操作

客户端收到数据后：

必须完全覆盖本地状态。

不得保留旧数据。

## MSG-019 HEARTBEAT（心跳）

消息方向：

客户端 ↔ 服务器

目的：

检测网络连接。

触发条件：

固定时间间隔。

请求：

```json
{
  "type": "HEARTBEAT"
}
```

服务器响应：

```json
{
  "type": "HEARTBEAT_ACK"
}
```

说明：

连续多个心跳失败。

服务器进入：

```text
DISCONNECTED
```

等待玩家重连。

达到超时时间。

服务器自动开启托管。

## MSG-020 ROOM_DISMISS（解散房间）

消息方向：

客户端 → 服务器

目的：

解散当前房间。

触发条件：

房主申请。

或：

玩家发起解散投票。

请求：

```json
{
  "type": "ROOM_DISMISS"
}
```

广播：

```json
{
  "type": "ROOM_DISMISS_VOTE",
  "applySeat": 0
}
```

若所有玩家同意：

服务器广播：

```json
{
  "type": "ROOM_DISMISS_SUCCESS"
}
```

下一状态：

ROOM_CLOSED

说明：

本项目采用：

全员同意解散。

若有人拒绝。

继续当前游戏。

若牌局已经结束。

服务器直接关闭房间。
